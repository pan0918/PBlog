# 照片墙内存优化设计

## 问题与证据

照片墙列表一次渲染每个相册的三张原始图片。实际浏览器采样中，15 张图片的原始尺寸主要为 3024×4096、4284×5712 和 5408×3680，而卡片显示尺寸约为 287×215～383。按 RGBA 解码估算，首屏图片占用约 921 MiB，实际显示像素只需约 5.2 MiB。打开相册后，每张新浏览的 4K 原图还会进入浏览器图片缓存。

## 目标

- 将相册列表和浏览器中的图片交给 Next.js Image Optimization 按视口输出。
- 保留现有布局、图片比例、清晰度、切换动画、键盘导航、触摸滑动、圆点导航和说明文字。
- 相册列表采用响应式缩略图；大图浏览器以当前视口需要的分辨率加载当前照片。
- 删除用于探测尺寸的原始 `new Image()` 请求，改为从已优化图片的 `onLoad` 事件读取比例。

## 实现

1. 在 `next.config.ts` 中移除全局 `unoptimized`，仅允许两个现有图片主机的 `/file/**` 路径，并允许质量 80、90。
2. `PhotoWallClient.tsx` 使用 `next/image` 的 `fill` 和 `sizes`：
   - 卡片背景图质量 80；
   - 保留三层照片堆叠效果；
   - 默认懒加载和异步解码。
3. `BookViewer.tsx` 使用质量 90 的响应式 `Image`：
   - 图片加载完成后根据优化后图片的宽高比计算与当前视口匹配的容器尺寸；
   - 加载期间保留现有骨架屏；
   - 每次只挂载当前幻灯片；
   - 不预载整本相册，避免再次扩大解码缓存。
4. 提取纯函数 `fitImageWithinViewport`，测试横图、竖图和无需缩放三种情况。

## 验证

- 自动测试确保远程主机白名单、响应式 Image、无原图探测和尺寸算法存在。
- ESLint 和生产构建必须通过。
- 浏览器验证首屏仍有五个三层相册卡片，并检查实际加载图片的自然尺寸显著低于原始 4K/5K 尺寸。
- 验证打开、关闭、前后翻页、键盘和触摸相关代码路径保持不变。
