# 公开作者资料设计

## 目标

博客前台统一将管理员作者显示为 `Frud_`，头像使用：

`https://a68b43cc.cloudflare-imgbed-9pz.pages.dev/file/1782456681130_圣诞猫猫.jpg`

后台登录账号、密码、管理员数据和登录流程保持不变，后台仍使用现有 `admin` 用户名登录。

## 方案

在站点配置中新增单一的公开作者资料配置，包含公开显示名和头像地址。公开作者资料只负责前台展示，不参与管理员身份验证。

所有前台作者身份出口统一读取该配置：

- `/api/auth/session` 检测到管理员会话时，返回公开显示名和公开头像。
- 评论仓储在映射管理员评论时，使用公开显示名和公开头像。
- 右上角账户入口、作者账户面板和评论组件继续消费统一的会话或评论作者数据，不分别硬编码资料。

## 数据流

管理员仍以 `admin` 登录并获得现有管理员 Cookie。服务端使用管理员 ID 判断作者身份；向前台返回作者信息时，将认证用户名替换为公开作者资料。评论数据仍通过 `admin_user_id` 识别作者，因此历史评论也会立即使用新的公开资料，不需要改写评论记录。

## 边界与错误处理

- 不修改 `admin_users` 表，不新增数据库迁移。
- 不修改管理员登录用户名、密码或 JWT 认证规则。
- 不允许普通读者通过注册获得作者身份。
- 公开头像加载失败时沿用浏览器图片失败行为，不影响登录和评论功能。

## 验证

- 测试公开作者配置的显示名和头像地址。
- 测试管理员会话接口使用公开作者资料，而不是 JWT 中的登录用户名。
- 测试评论仓储对管理员评论使用公开作者资料，并保留 `isAuthor: true`。
- 运行全量测试、TypeScript、ESLint 和生产构建。
